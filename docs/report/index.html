一

项目背景与问题定义

1.1 医疗随访的现实困境

在基层医疗场景中，患者就诊后的"诊后断联"现象是一个长期被忽视却影响深远的系统性问题。

患者维度：大多数患者在取药离诊后不会主动复诊或反馈治疗效果。数据显示，基层门诊的自然复诊率不足 30%，意味着超过 70% 的患者就诊结果对医生而言是完全黑盒的。

医生维度：基层全科医生每日接诊量通常在 30 至 50 人次，繁重的日常工作使得主动随访几乎不可能实现。即便有意愿，也缺乏系统化工具支持。

医术成长维度：临床医术的精进依赖"诊断→治疗→反馈→调整"的闭环。当反馈环节几乎不存在时，医生只能依赖有限的显性案例积累经验，难以系统性地评估自己的用药方案是否最优、诊断思路是否准确。

1.2 核心痛点总结

痛点影响
患者复诊率极低医生无法获知治疗效果
随访工作全靠人力医生没有时间和精力执行
治疗反馈断链无法积累数据优化用药策略
依从性不透明无法区分"药不对症"和"患者未遵医嘱"
高危患者缺乏预警可能错过最佳干预时机
二

解决方案：PatientClaw

2.1 核心思路

PatientClaw 是一个基于 OpenClaw（本地 AI 智能体框架）构建的患者诊后智能随访系统。其核心思路是：

用 AI 全自动承担随访工作，医生只需查看日报，无需任何额外操作。

系统在医生完全不改变工作习惯的前提下，自动模拟患者随访全流程：生成随访消息、模拟患者回复、解析回复数据，最终将洞察以结构化日报的形式推送给医生。

2.2 角色分工

角色由谁扮演说明
医生（李晓峰）真实用户只需被动接收飞书日报
患者（P001-P030）OpenClaw 模拟自动生成回复，模拟真实延迟和内容

2.3 工作流程

                    ↓

三

系统功能详解

3.1 智能个性化随访消息

区别于通用模板，PatientClaw 的随访消息由 OpenClaw 结合以下上下文动态生成：

  • 患者的具体诊断结果
  • 所用药物及服用方式
  • 就诊已过天数（Day 3 / 7 / 14 问法不同）
  • 本患者的随访历史（避免重复提问）
示例 · 王建国，高血压，Day 3
您好，王建国先生！我是李晓峰主任医师诊室的智能随访助手。您上次就诊后已经过去3天了，氨氯地平片是否已经开始规律服用？这几天血压自测情况怎么样？有没有出现踝部水肿或头痛加重的情况？请简单告知，医生会及时了解您的情况。
示例 · 韩梅，心力衰竭，Day 7
韩阿姨您好！距上次李晓峰主任调整您的心衰方案（加用沙库巴曲缬沙坦）已一周了，新方案服用是否顺利？这几天气短和双踝水肿情况有没有改善？每天早晨的体重记录怎么样？您的情况对医生评估用药效果非常重要，期待您的回复。

3.2 患者回复智能解析

OpenClaw 从患者的自然语言回复中提取结构化信息，并自动判断优先级：

{
  "medication_adherence": "good",
  "condition_status": "improved",
  "adverse_reaction": "轻微踝部水肿",
  "needs_followup": true,
  "priority_level": "attention",
  "summary": "患者按时服药，血压下降至138/88mmHg，出现轻度踝部水肿，需关注"
}

优先级判断标准：urgent 胸痛加重/呼吸困难/晕厥   attention 依从性差/症状无改善   normal 用药正常/症状改善

3.3 个人病情追踪报告

每位患者在 Day 14 随访后触发生成，包含：

  • 基本信息与诊断摘要
  • 药方详情
  • 三次随访记录时间线
  • 用药依从性评估
  • 病情进展趋势分析
  • 后续建议

3.4 医生综合日报（每天）

每天自动生成并推送，采用"重点优先"结构：

  1. ⚠️ 需重点关注的患者（未痊愈 / 症状加重 / 依从性差）
  2. ✅ 情况良好 / 已痊愈的患者
  3. 📊 用药方案效果统计
  4. 🔔 建议主动联系或安排复诊的患者
四

技术实现

4.1 技术栈

组件方案
AI 推理与编排OpenClaw（本地部署）
消息通道飞书机器人（Feishu Bot API）
数据存储CSV 文件 + Markdown 病情档案（data/reports/）
任务调度三类定时任务均每 10 分钟触发一次（= 系统内 1 天）

4.2 数据模型

文件核心字段
patients.csvpatient_id, name, age, gender, feishu_account, medical_history, allergies
visits.csvvisit_id, patient_id, doctor_id, visit_date, diagnosis, notes
prescriptions.csvprescription_id, visit_id, medication, dosage, frequency, duration_days
followups.csvfollowup_id, patient_id, followup_day, medication_adherence, condition_status, priority_level, ai_summary

4.3 核心 Prompt 设计原则

  • 随访消息：温暖专业，结合诊断定制问题，150字以内，引导简短回复
  • 回复解析：输出固定 JSON 结构，不确定项标记 unknown，不猜测
  • 医生报告：重点突出，数据驱动，每条建议可操作

依从性与异常处理

患者回复延迟模型

系统模拟真实患者的回复行为，包含四种延迟类型：

40%
本轮回复
随访触发时立即生成回复
30%
下轮才回复
下次随访触发时补回复
20%
再下轮才回复
再等一轮后生成回复
10%
不回复
超过3次触发后标记 no_reply

真实问题案例

系统在 10 天演示中自动识别并标记了以下真实异常案例：

P010 吴霞 attention
高血压患者自行将比索洛尔改为隔日服用，血压反弹至 134/86 mmHg，心率升至 82 bpm，焦虑症状回升。系统自动标记依从性差，提醒医生关注。
P019 段勇 attention
冠心病支架术后患者因胃部不适，阿司匹林漏服 3 次。系统识别到单抗中断风险，标记为需关注，建议医生评估是否需要加用胃黏膜保护剂。
P029 邓凯 urgent
重度心衰患者（EF 30%），随访发现体重较上次增加 1.6 kg，气短症状无改善。系统自动升级为 urgent 预警，立即通知医生，建议评估是否需要调整利尿剂剂量。
P012 马超 attention
CKD 3 期合并高血压患者，随访中反映近期乏力明显，系统结合其肾功能背景标记为需关注，建议复查肾功能及电解质。
六

项目价值与影响

对医生：从"经验积累靠感觉"到"数据驱动的临床反思"，低成本获得过去几乎不可能拥有的随访数据。

对患者：每位患者感受到被关注，同时获得用药提醒和专业问询，提升治疗依从性和满意度。

对医疗系统：随访的自动化降低了医疗资源压力，同时为早期识别高危患者提供了可能。

可扩展性

  • 支持接入电子病历系统（替换 CSV 数据层）
  • 支持更多消息渠道（微信、短信）
  • 支持多医生、多科室模式
  • 可积累数据后进行横向用药效果分析
