# PatientClaw：患者诊后病情全自动跟踪统计系统

**项目说明书** · 北纬·龙虾大赛（第一届）· OpenClaw Hackathon · 生产力赛道

---

## 一、项目背景与问题定义

### 1.1 医疗随访的现实困境

在基层医疗场景中，患者就诊后的"诊后断联"现象是一个长期被忽视却影响深远的系统性问题。

**患者维度**：大多数患者在取药离诊后不会主动复诊或反馈治疗效果。数据显示，基层门诊的自然复诊率不足30%，意味着超过70%的患者就诊结果对医生而言是完全黑盒的。

**医生维度**：基层全科医生每日接诊量通常在30至50人次，繁重的日常工作使得主动随访几乎不可能实现。即便有意愿，也缺乏系统化工具支持。

**医术成长维度**：临床医术的精进依赖"诊断→治疗→反馈→调整"的闭环。当反馈环节几乎不存在时，医生只能依赖有限的显性案例积累经验，难以系统性地评估自己的用药方案是否最优、诊断思路是否准确。

### 1.2 核心痛点总结

| 痛点 | 影响 |
|------|------|
| 患者复诊率极低 | 医生无法获知治疗效果 |
| 随访工作全靠人力 | 医生没有时间和精力执行 |
| 治疗反馈断链 | 无法积累数据优化用药策略 |
| 依从性不透明 | 无法区分"药不对症"和"患者未遵医嘱" |
| 高危患者缺乏预警 | 可能错过最佳干预时机 |

## 二、解决方案：PatientClaw

### 2.1 核心思路

PatientClaw 是一个基于 **OpenClaw**（本地 AI 智能体框架）构建的患者诊后智能随访系统。其核心思路是：

> **我用 OpenClaw 龙虾全自动承担随访工作，医生只需查看日报，无需任何额外操作。**

系统在医生完全不改变工作习惯的前提下，由龙虾自动完成患者随访全流程：生成随访消息、模拟患者回复、解析回复数据，最终将洞察以结构化日报的形式推送给医生。

### 2.2 角色分工

| 角色 | 由谁扮演 | 说明 |
|------|---------|------|
| 医生（李晓峰） | 真实用户 | 只需被动接收飞书日报 |
| 患者（P001-P030） | OpenClaw 模拟 | 自动生成回复，模拟真实延迟和内容 |

### 2.3 工作流程

```
就诊 (Day 0)
  ↓
  Day 3 ─── OpenClaw 生成个性化随访消息（首次随访，就诊后30分钟触发）
                    ↓
            OpenClaw 模拟患者回复 → 解析回复 → 更新病情档案
                    ↓
  Day 7 ─── 同上（中期随访，70分钟触发）
                    ↓
  Day 14 ── 同上（末期随访，140分钟触发）→ 生成完整版病情追踪报告
                                            ↓
  每10分钟 ── OpenClaw 汇总分析 ──────────→ 医生综合日报（飞书推送）
```

### 2.4 OpenClaw 能力映射

| 功能模块 | 使用的 OpenClaw 能力 |
|---------|-------------------|
| 定时触发随访 | 调度引擎（定时任务） |
| 个性化消息生成 | LLM 推理 + 模板引擎 |
| 飞书消息收发 | 飞书 Bot 集成（Tool Use） |
| 患者回复理解 | LLM NLP 解析 + JSON 结构化输出 |
| 随访数据写入 | 文件操作（CSV 读写） |
| 报告生成 | LLM 生成 + 格式化输出 |

## 三、系统功能详解

### 3.1 智能个性化随访消息

区别于通用模板，PatientClaw 的随访消息由 OpenClaw 结合以下上下文动态生成：
- 患者的具体诊断结果
- 所用药物及服用方式
- 就诊已过天数（Day 3 / 7 / 14 问法不同）
- 本患者的随访历史（避免重复提问）

**真实示例（P005 陈强，心力衰竭，Day 7）：**
> 陈强您好！我是李晓峰主任诊室的随访助理。您就诊已满一周，请问这几天沙库巴曲缬沙坦、美托洛尔、螺内酯、达格列净是否每日按时服用？今日体重是多少？有无气短复发或双踝水肿？血压今天测了多少？心衰患者体重每日变化很重要，如近两天体重增加超过2kg请及时告知。请简要回复。

**患者回复：**
> 药都按时吃了，今天体重62.8kg，血压128/80，没有气短，脚也不肿，感觉比上周好多了，走路轻松了不少。

### 3.2 患者回复智能解析

OpenClaw 从患者的自然语言回复中提取结构化信息，并自动判断优先级：

```json
{
  "medication_adherence": "good",
  "condition_status": "improved",
  "adverse_reaction": null,
  "needs_followup": false,
  "priority_level": "normal",
  "summary": "体重62.8kg稳定，血压128/80，无气短水肿，心衰症状持续改善，用药依从性好。"
}
```

优先级判断标准：
- `urgent`：胸痛加重 / 呼吸困难 / 晕厥 / 严重不适 → 立即通知医生
- `attention`：依从性差 / 症状无改善 / 出现不良反应
- `normal`：用药正常 / 症状改善 / 无明显不适

### 3.3 患者回复延迟模型

系统模拟真实患者的回复行为，包含四种延迟类型：

| 类型 | 概率 | 说明 |
|------|------|------|
| same_round | 40% | 随访触发时立即生成回复 |
| next_round | 30% | 下次随访触发时补回复 |
| later_round | 20% | 再等一轮后生成回复 |
| no_reply | 10% | 超过3次触发后标记为未回复 |

### 3.4 个人病情追踪报告

每位患者持续维护独立档案，Day 14 后生成完整版终期报告，包含：
- 基本信息与诊断摘要
- 药方详情
- 随访时间线（每次随访一行）
- 用药依从性评估
- 病情进展趋势分析
- 后续随访建议

### 3.5 医生综合日报（每天）

每天自动生成并推送，采用"重点优先"结构：
1. ⚠️ 需立即关注的患者（症状加重 / 依从性差 / 高危预警）
2. 🔔 需持续关注的患者
3. ✅ 今日状态良好的患者
4. 📭 未回复患者
5. 📊 用药方案效果统计
6. 💡 今日用药洞察与建议

## 四、技术实现

### 4.1 技术栈

| 组件 | 方案 |
|------|------|
| AI 推理与编排 | OpenClaw（本地部署） |
| 消息通道 | 飞书机器人（Feishu Bot API） |
| 数据存储 | CSV 文件 + Markdown 病情档案（data/reports/） |
| 任务调度 | OpenClaw 内置定时任务（三类任务均每 10 分钟触发） |

### 4.2 时间压缩模型

为在演示中完整呈现随访全流程，系统采用时间压缩设计：**系统内每过 10 分钟 = 现实中过了 1 天**

| 系统时间 | 现实含义 | 触发事件 |
|---------|---------|---------|
| +30 分钟 | 就诊后第 3 天 | 首次随访询问 |
| +70 分钟 | 就诊后第 7 天 | 中期随访询问 |
| +140 分钟 | 就诊后第 14 天 | 末期随访 + 完整报告生成 |
| 每 10 分钟 | 每天 | 医生日报生成 + 推送 |

### 4.3 数据模型

| 文件 | 核心字段 |
|------|---------|
| `patients.csv` | patient_id, name, age, gender, feishu_account, primary_condition, medical_history, allergies |
| `visits.csv` | visit_id, patient_id, doctor_id, visit_date, visit_num, diagnosis, bp, hr, notes |
| `prescriptions.csv` | prescription_id, visit_id, medication, dosage, frequency, duration_days, instructions |
| `followups.csv` | followup_id, patient_id, visit_id, followup_day, scheduled_time, sent_time, message_sent, reply_received, reply_delay_type, medication_adherence, condition_status, adverse_reaction, priority_level, ai_summary |

### 4.4 核心 Prompt 设计原则

- **随访消息**：温暖专业，结合诊断定制问题，150字以内，引导简短回复
- **回复解析**：输出固定 JSON 结构，不确定项标记 unknown，不猜测
- **医生报告**：重点突出，数据驱动，每条建议可操作

## 五、演示数据与真实案例

### 5.1 数据集规模

| 指标 | 数值 |
|------|------|
| 患者总数 | 30 人 |
| 随访记录 | 50 条 |
| 完整日报 | 10 天（2026-03-18 至 2026-03-27） |
| 病情追踪报告 | 30 份 |
| 整体用药依从率 | 约 92% |

### 5.2 病种分布

| 病种 | 患者数 | 代表患者 |
|------|--------|---------|
| 高血压（含合并症） | 13 人 | P001-P013 |
| 冠心病 / 心绞痛 | 8 人 | P014-P021 |
| 心律失常 | 4 人 | P022-P025 |
| 心力衰竭 | 5 人 | P026-P030 |

### 5.3 真实异常案例

系统在 10 天演示中自动识别并标记了以下真实异常案例：

**P010 吴霞（高血压合并焦虑）— attention**
自行将比索洛尔改为隔日服用，血压反弹至 134/86 mmHg，心率升至 82 bpm，焦虑症状回升。系统自动标记依从性差，提醒医生关注。

**P019 段勇（冠心病支架术后）— attention**
因胃部不适，阿司匹林漏服 3 次。系统识别到单抗中断风险，建议医生评估是否加用胃黏膜保护剂。

**P029 邓凯（重度心衰 EF 30%）— attention**
体重较就诊增加 1.6 kg，气短症状无改善。系统自动升级关注级别，建议评估是否调整利尿剂剂量。

**P012 马超（高血压 / 房颤 / CKD 3期）— attention**
随访中反映乏力明显，系统结合其肾功能背景标记为需关注，建议复查肾功能及电解质。

## 六、项目价值与影响

**对医生**：从"经验积累靠感觉"到"数据驱动的临床反思"，低成本获得过去几乎不可能拥有的随访数据。

**对患者**：每位患者感受到被关注，同时获得用药提醒和专业问询，提升治疗依从性和满意度。

**对医疗系统**：随访的自动化降低了医疗资源压力，同时为早期识别高危患者提供了可能。

**可扩展性**：
- 支持接入电子病历系统（替换 CSV 数据层）
- 支持更多消息渠道（微信、短信）
- 支持多医生、多科室模式
- 可积累数据后进行横向用药效果分析
