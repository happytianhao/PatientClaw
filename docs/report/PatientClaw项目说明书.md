## 一、项目背景与社会价值

### 1.1 一个被长期忽视的社会问题：患者诊后失联

在基层医疗场景中，患者就诊后的"诊后断联"现象是一个影响深远却几乎无人解决的社会问题。

**数据说话：**
- 基层门诊自然复诊率不足 **30%**，超过 **70%** 的患者离诊后音讯全无
- 医生无法获知治疗效果，医术成长依赖的"诊断→治疗→反馈→调整"闭环几乎不存在
- 心血管疾病是需要持续管理的慢性病，患者用药依从性差是全球性难题

**三类受害群体：**

| 群体 | 现状 | 困境 |
|------|------|------|
| 患者 | 离诊后无人问津 | 有问题不知道找谁，小病拖成大病 |
| 医生 | 工作超负荷 | 想知道效果但没时间随访，只能靠猜 |
| 医疗系统 | 资源紧张 | 本可预防的并发症因失联而演变成高昂的急诊/住院费用 |

**这不是技术问题，是系统性问题。** 没有合适工具，没有合适机制，没有人负责。PatientClaw 用 AI 技术填补了这个空白。

### 1.2 核心价值主张

> **PatientClaw 让每一次就诊都有后续。**

这不是一个炫技的项目——它解决的是一个真实存在的、影响数以亿计慢性病患者的系统性问题。用 AI 把随访工作自动化，让医生看病有数据支撑，让患者康复有专业守护。

## 二、解决方案：PatientClaw

### 2.1 核心思路

PatientClaw 是一个基于 **OpenClaw**（本地 AI 智能体框架）构建的患者诊后智能随访系统。其核心思路是：

> **用 OpenClaw 龙虾全自动承担随访工作，医生只需查看日报，无需任何额外操作。**

系统在医生完全不改变工作习惯的前提下，由龙虾自动完成患者随访全流程：生成随访消息、模拟患者回复、解析回复数据，最终将洞察以结构化日报的形式推送给医生。

### 2.3 工作流程

```
就诊 (Day 0)
  ↓

                    ↓
            OpenClaw 模拟患者回复 → 解析回复 → 更新病情档案
                    ↓

                                            ↓
  每10分钟 ── OpenClaw 汇总分析 ──────────→ 医生综合日报（飞书推送）
```

## 三、应用价值详解

### 3.1 对社会的价值

**降低医疗成本：** 良好的随访管理可将心血管事件再发率降低 20-30%。假设一名心梗患者的急诊+住院费用约 5-10 万元，系统每预防一次事件即可节省大量医保和个人支出。

**提升基层医疗质量：** 基层医生最大的短板是没有反馈，不知道自己的诊疗方案是否最优。PatientClaw 让每一位基层医生都拥有了过去只有顶级医院才有的随访体系。

**改善医患关系：** 患者感受到被持续关注，满意度自然提升，治疗依从性也更好——这是一个正向循环。

**数据驱动的临床研究：** 系统积累的真实世界随访数据，可用于药物效果比较、临床路径优化等公共卫生研究。

### 3.2 对患者的具体关怀

PatientClaw 不只是发消息，它通过三个维度的关怀，让患者真正感受到"有人在管我"：

- **专业性**：每次随访都基于患者的具体诊断和用药，不是通用模板，让患者感受到医生的认真负责
- **及时性**：在正确的时间发出问询（Day 3 / 7 / 14），不错过任何一个关键节点
- **预警性**：发现问题的患者会立即升级到 urgent，医生的及时介入可能挽救一条生命

### 3.3 可扩展性路径

PatientClaw 的设计天然支持规模扩展：

| 扩展方向 | 当前基础 | 扩展路径 |
|---------|---------|---------|
| 科室扩展 | 心血管科 30 人 | 内分泌科（糖尿病）、肿瘤科、精神科等慢性病管理 |
| 地域扩展 | 单诊所模拟 | 县级医院、社区卫生中心、家庭医生签约服务 |
| 渠道扩展 | 飞书机器人 | 微信、短信、电话，覆盖无智能手机用户 |
| 系统集成 | CSV 数据层 | 接入电子病历（EMR）、医保系统、家庭医生签约平台 |
| 数据应用 | 30人演示数据 | 横向用药效果分析、临床路径优化、公共卫生研究 |

## 四、系统功能详解

### 4.1 智能个性化随访消息

区别于通用模板，PatientClaw 的随访消息由 OpenClaw 结合以下上下文动态生成：
- 患者的具体诊断结果
- 所用药物及服用方式
- 就诊已过天数（Day 3 / 7 / 14 问法不同）
- 本患者的随访历史（避免重复提问）

**真实示例（P005 陈强，心力衰竭，Day 7）：**
> 陈强您好！我是李晓峰主任诊室的随访助理。您就诊已满一周，请问这几天沙库巴曲缬沙坦、美托洛尔、螺内酯、达格列净是否每日按时服用？今日体重是多少？有无气短复发或双踝水肿？血压今天测了多少？心衰患者体重每日变化很重要，如近两天体重增加超过2kg请及时告知。请简要回复。

**患者回复：**
> 药都按时吃了，今天体重62.8kg，血压128/80，没有气短，脚也不肿，感觉比上周好多了，走路轻松了不少。

### 4.2 患者回复智能解析

OpenClaw 从患者的自然语言回复中提取结构化信息，并自动判断优先级：

```json
{
  "medication_adherence": "good",
  "condition_status": "improved",
  "adverse_reaction": null,
  "needs_followup": false,
  "priority_level": "normal",
  "summary": "体重62.8kg稳定，血压128/80，无气短水肿，心衰症状持续改善，用药依从性好。"
}
```

优先级判断标准：
- `urgent`：胸痛加重 / 呼吸困难 / 晕厥 / 严重不适 → 立即通知医生
- `attention`：依从性差 / 症状无改善 / 出现不良反应
- `normal`：用药正常 / 症状改善 / 无明显不适

### 4.3 患者回复延迟模型

系统模拟真实患者的回复行为，包含四种延迟类型：

| 类型 | 概率 | 说明 |
|------|------|------|
| same_round | 40% | 随访触发时立即生成回复 |
| next_round | 30% | 下次随访触发时补回复 |
| later_round | 20% | 再等一轮后生成回复 |
| no_reply | 10% | 超过3次触发后标记为未回复 |

### 4.4 个人病情追踪报告

每位患者持续维护独立档案，Day 14 后生成完整版终期报告，包含：
- 基本信息与诊断摘要
- 药方详情
- 随访时间线（每次随访一行）
- 用药依从性评估
- 病情进展趋势分析
- 后续随访建议

### 4.5 医生综合日报（每天）

每天自动生成并推送，采用"重点优先"结构：
1. ⚠️ 需立即关注的患者（症状加重 / 依从性差 / 高危预警）
2. 🔔 需持续关注的患者
3. ✅ 今日状态良好的患者
4. 📭 未回复患者
5. 📊 用药方案效果统计
6. 💡 今日用药洞察与建议

## 五、技术实现

### 5.1 技术栈

| 组件 | 方案 |
|------|------|
| AI 推理与编排 | OpenClaw（本地部署） |
| 消息通道 | 飞书机器人（Feishu Bot API） |
| 数据存储 | CSV 文件 + Markdown 病情档案（data/reports/） |
| 任务调度 | OpenClaw 内置定时任务（三类任务均每 10 分钟触发） |

### 5.2 时间压缩模型

为在演示中完整呈现随访全流程，系统采用时间压缩设计：**系统内每过 10 分钟 = 现实中过了 1 天**

| 系统时间 | 现实含义 | 触发事件 |
|---------|---------|---------|
| +30 分钟 | 就诊后第 3 天 | 首次随访询问 |
| +70 分钟 | 就诊后第 7 天 | 中期随访询问 |
| +140 分钟 | 就诊后第 14 天 | 末期随访 + 完整报告生成 |
| 每 10 分钟 | 每天 | 医生日报生成 + 推送 |

### 5.3 数据模型

| 文件 | 核心字段 |
|------|---------|
| `patients.csv` | patient_id, name, age, gender, feishu_account, primary_condition, medical_history, allergies |
| `visits.csv` | visit_id, patient_id, doctor_id, visit_date, visit_time, clinic_location, visit_num, chief_complaint, diagnosis, bp, hr, examinations, exam_results, notes |
| `prescriptions.csv` | prescription_id, visit_id, medication, dosage, frequency, duration_days, instructions |
| `followups.csv` | followup_id, patient_id, visit_id, followup_day, scheduled_time, sent_time, message_sent, reply_received, reply_delay_type, medication_adherence, condition_status, adverse_reaction, priority_level, ai_summary |

### 5.4 核心 Prompt 设计原则

- **随访消息**：温暖专业，结合诊断定制问题，150字以内，引导简短回复
- **回复解析**：输出固定 JSON 结构，不确定项标记 unknown，不猜测
- **医生报告**：重点突出，数据驱动，每条建议可操作

## 七、项目价值与影响

**对医生**：从"经验积累靠感觉"到"数据驱动的临床反思"，低成本获得过去几乎不可能拥有的随访数据。

**对患者**：每位患者感受到被关注，同时获得用药提醒和专业问询，提升治疗依从性和满意度。

**对医疗系统**：随访的自动化降低了医疗资源压力，同时为早期识别高危患者提供了可能。

**可扩展性**：
- 支持接入电子病历系统（替换 CSV 数据层）

- 支持多医生、多科室模式
- 可积累数据后进行横向用药效果分析
