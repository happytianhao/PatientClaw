# PatientClaw × OpenClaw 配置教程

**手把手指南 · 从零开始让你的龙虾跑起来**

> 本教程假设你已完成 OpenClaw 本地安装，手把手带你配置 PatientClaw 的全自动随访系统。

---

## 第一步：理解系统要做什么

在动手之前，先明确整个系统的工作流程：

```
每隔 10 分钟（= 系统里的1天）：
  龙虾 → 检查今天哪些患者需要随访
  龙虾 → 根据患者的诊断和药方，生成专属随访消息
  龙虾 → 模拟患者回复（含延迟/不回复逻辑）
  龙虾 → 分析回复，提取用药和病情信息，写入数据文件
  龙虾 → 每 10 分钟（= 1天），生成医生日报发到飞书
  医生 → 在飞书查看日报（唯一需要人工参与的环节）
```

## 第二步：准备飞书机器人

> **单账号说明**：本系统只需要一个真实飞书账号（医生账号）。患者侧的所有消息收发均由 OpenClaw 在内部模拟，不需要真实患者账号。你只需创建一个飞书应用机器人，用于向医生账号推送日报。

### 2.1 创建企业自建应用

1. 打开飞书开放平台：https://open.feishu.cn/app
2. 点击「**创建企业自建应用**」
3. 填写应用信息：
   - 应用名称：`PatientClaw 随访助手`
   - 应用描述：`心血管科患者诊后智能随访系统`
4. 创建成功后，进入应用详情页，记录以下信息（后面要用）：
   - `App ID`（应用 ID）
   - `App Secret`（应用密钥）

### 2.2 开启机器人能力

在应用管理页面，进入「**应用能力**」→「**机器人**」，点击开启。

### 2.3 申请 API 权限

在左侧导航栏进入「**开发配置**」→「**权限管理**」，点击「**开通权限**」，搜索并开通以下权限：

| 权限 | 用途 |
|------|------|
| `im:message` | 收发消息 |
| `im:message:send_as_bot` | 以机器人身份发送消息 |
| `im:resource` | 下载消息资源（图片等） |

### 2.4 配置事件订阅（接收消息）

在左侧导航栏进入「**开发配置**」→「**事件与回调**」→「**事件订阅**」：

1. 请求地址填入 bridge 的 webhook 地址（第三步配置后获得）：
   ```
   http://你的服务器IP:3000/webhook
   ```
2. 添加事件：搜索 `im.message.receive_v1`（接收消息），点击添加

### 2.5 发布应用

1. 在左侧导航栏进入「**应用发布**」→「**版本管理与发布**」，点击「**创建版本**」
2. 填写版本号（如 `1.0.0`）和更新说明
3. 点击「**申请线上发布**」
4. 等待企业管理员审核通过（自己是管理员可直接通过）

> **测试阶段免审技巧**：在「**测试企业与人员**」中添加自己的账号，切换为测试版本后，权限无需审核即可生效，方便本地调试。

## 第三步：在 OpenClaw 中配置飞书连接

### 3.1 安装 lark-integration skill

在 OpenClaw 中安装飞书集成 skill：

```bash
# 在 OpenClaw 的 skills 目录下已内置 lark-integration
# 确认 skills/lark-integration/scripts/bridge-webhook.mjs 存在
```

### 3.2 配置凭证

```bash
# 写入 App ID
echo "FEISHU_APP_ID=cli_你的AppID" >> ~/.openclaw/workspace/.env

# 写入 App Secret（单独存放，更安全）
mkdir -p ~/.openclaw/secrets
echo "你的AppSecret" > ~/.openclaw/secrets/feishu_app_secret
```

### 3.3 启动 webhook bridge

```bash
cd skills/lark-integration/scripts
node bridge-webhook.mjs
```

启动成功后，bridge 会监听 `http://localhost:3000/webhook`。

> 如果服务器有公网 IP，把 `localhost` 换成公网 IP，填入第二步 2.4 的事件订阅地址。

### 3.4 创建 OpenClaw 项目

1. 打开 OpenClaw Web 界面
2. 点击「**新建项目**」或「**New Agent**」
3. 项目名称：`PatientClaw`
4. 工作目录指向：PatientClaw 项目根目录

## 第四步：配置核心 Agent（随访任务）

### 4.1 创建主调度 Agent

在 OpenClaw 中新建一个 Agent，命名为 `随访调度器`。

**系统提示词（System Prompt）**：

```
你是北京协和医院心血管内科李晓峰主任医师的智能随访助理——PatientClaw。

你的职责：
1. 每次被激活时，读取项目 data/ 目录下的患者数据
2. 判断哪些患者今天需要随访（根据就诊日期和系统时间）
3. 为需要随访的患者生成个性化随访消息并通过飞书发送
4. 记录所有操作到 chat_logs.csv 和 followups.csv

时间规则：
- 系统内每过10分钟 = 现实中过了1天
- 就诊后第3天、第7天、第14天需要随访
- 如果患者3天（30分钟）未回复，标记为 no_reply 并提升优先级

你必须：
- 阅读患者的具体诊断和用药后再生成消息（不能用通用模板）
- 保持专业且温暖的语气
- 所有操作都要写入对应的 CSV 文件留存记录
```

**配置定时触发**：
- 触发方式：定时（Scheduled）
- 频率：**每 10 分钟执行一次**
- 时区：Asia/Shanghai

### 4.2 配置随访消息生成工具

在 Agent 的工具列表中，添加以下工具调用：

**工具1：读取患者数据**
```
工具名称：read_patient_data
描述：读取指定患者的完整信息（基本信息+就诊记录+药方+历次随访）
输入：patient_id
输出：患者完整数据对象
实现：读取 patients.csv + visits.csv + prescriptions.csv + followups.csv
```

**工具2：发送飞书消息**
```
工具名称：send_feishu_message
描述：向指定飞书账号发送消息
输入：feishu_account（账号）, message（消息内容）
输出：发送成功/失败状态
```

**工具3：写入随访记录**
```
工具名称：write_followup_record
描述：将随访信息写入 followups.csv
输入：followup数据对象（见数据结构设计）
输出：写入成功/失败
```

**工具4：写入聊天记录**
```
工具名称：append_chat_log
描述：向 chat_logs.csv 追加一条记录
输入：patient_id, sender, message_type, content
输出：写入成功
```

## 第五步：配置随访消息生成 Prompt

这是系统最核心的部分。在 OpenClaw 中创建一个「**Prompt 模板**」：

**模板名称**：`followup_message_template`

**Prompt 内容**：

```
你是北京协和医院心血管内科李晓峰主任医师的智能随访助理。

当前任务：为以下患者生成第{day}天随访消息。

【患者信息】
姓名：{name}，{age}岁，{gender}，{occupation}
就诊日期：{visit_date}
主诉：{chief_complaint}
诊断：{diagnosis}
血压/心率：{bp}/{hr}
医嘱要点：{notes}

【当前用药】
{medications_list}

【历次随访摘要】
{previous_followups_summary}

【生成要求】
1. 语气温暖专业，像医生助理（不是医生本人）在关心患者
2. 针对该患者的具体病情提问：
   - 高血压患者：必问血压数值
   - 冠心病患者：必问胸痛/胸闷情况
   - 心衰患者：必问气短/水肿/体重变化
   - 心律失常患者：必问心悸/心跳不规律感觉
   - 服用抗凝药患者：必问有无异常出血
3. 第{day}天随访重点：
   - 第3天：主要问是否已开始规律服药，有无初期不适
   - 第7天：主要问症状是否改善，用药是否规律，有无副作用
   - 第14天：综合评估，问是否需要复诊，整体状态如何
4. 字数120-150字，结尾明确提示患者回复什么内容
5. 不得出现与该患者诊断无关的问题
6. 不得给出用药建议或修改医嘱

只输出消息正文，不要有任何额外说明。
```

## 第六步：配置患者回复处理

### 6.1 创建回复监听 Agent

新建 Agent，命名为 `回复处理器`。

**触发方式**：消息触发（Message Received）——飞书收到患者消息时自动激活

**系统提示词**：

```
你是一个医疗数据结构化助手。

当你收到患者通过飞书发来的消息时：
1. 查找该飞书账号对应的患者（在 patients.csv 中查 feishu_account 字段）
2. 找到该患者最近一次待回复的随访记录（followups.csv 中 reply_received 为空的最新记录）
3. 分析患者的回复，提取关键医疗信息
4. 更新 followups.csv 对应记录（填入回复内容和分析结果）
5. 将回复写入 chat_logs.csv
6. 如果发现紧急情况（胸痛加重/严重气短/晕厥等），立即通过飞书向医生发送提醒

分析患者回复时，必须输出以下结构化数据并写入 followups.csv：
- medication_adherence: good/partial/none/unknown
- condition_status: improved/stable/unchanged/worsened/recovered
- adverse_reaction: 不良反应描述（无则为null）
- priority_level: normal/attention/urgent
- ai_summary: 50字以内摘要
```

### 6.2 配置紧急情况提醒

在「回复处理器」的逻辑中，当 `priority_level = urgent` 时，触发以下操作：

```
立即发送飞书消息给医生（lixiaofeng_doctor@example.com）：

🚨 紧急患者提醒

患者：{name}（{patient_id}）
诊断：{diagnosis}
回复内容：{reply_text}
问题：{attention_reason}

建议立即联系患者或安排急诊就诊。
```

## 第七步：配置患者回复模拟（演示用）

由于这是演示系统，需要模拟患者回复。新建 Agent：`患者回复模拟器`

**系统提示词**：

```
你是一个医疗场景模拟助手。

你的任务：模拟患者收到随访消息后的真实回复。

当你需要模拟一个患者的回复时，请根据以下信息生成：

患者信息：{patient_info}
随访消息：{followup_message}
患者性格/背景：根据职业和年龄推断（老年退休患者更配合，年轻职场人可能更简短）

模拟规则：
1. 回复延迟模拟（从随机中选一种）：
   - 立即回复（30%）：2-5分钟后
   - 延迟回复（40%）：30-120分钟后
   - 长时延迟（20%）：200-400分钟后
   - 不回复（10%）：超过300分钟后标记 no_reply

2. 内容真实性（根据患者病情当前阶段随机）：
   - 60%的情况：病情按预期改善，用药基本规律
   - 20%的情况：用药依从性差（忘吃/自行减量）
   - 10%的情况：出现轻微不良反应
   - 5%的情况：症状加重需要关注
   - 5%的情况：病情明显好转/痊愈

3. 语言风格：
   - 老年患者（60+）：更详细、更配合、用词朴实
   - 中年患者（40-60）：较简洁、时间紧迫感
   - 年轻患者（<40）：简短、有时用网络语

只输出患者的回复文字，不要有任何额外说明。
```

**触发方式**：每次随访消息发送后，延迟随机时间自动触发此 Agent 生成回复。

## 第八步：配置医生每日报告

新建 Agent：`医生日报生成器`

**触发方式**：定时，每 10 分钟（= 系统内1天）触发一次

**系统提示词**：

```
你是一个医疗数据分析助手。

当被触发时，执行以下操作：
1. 读取过去10分钟内（= 过去1天）的所有随访记录（followups.csv）
2. 读取所有患者当前状态（patients.csv + 最新followups）
3. 读取所有聊天记录（chat_logs.csv）
4. 分析汇总，生成医生综合日报

报告格式（严格按此结构输出）：

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PatientClaw 日报
李晓峰主任医师 · {date}（系统第{system_day}天）
今日随访更新：{total}人 · 在管患者总计：{all}人
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

⚠️ 需立即关注（{n}人）
──────────────────────
{urgent_patients_list}
每人格式：• {name}（{diagnosis}）- {problem} - 建议：{action}

🔔 需持续关注（{n}人）
──────────────────────
{attention_patients_list}

✅ 今日状态良好（{n}人）
──────────────────────
{good_patients_table}
格式：姓名 | 诊断 | 用药依从 | 症状变化

📭 未回复患者（{n}人）
──────────────────────
{no_reply_list}
格式：姓名 | 诊断 | 最后随访日期 | 等待天数

📊 用药方案效果（累计）
──────────────────────
{drug_efficacy_table}
格式：药物名称 | 使用患者数 | 平均依从率 | 症状改善率 | 不良反应发生率

💡 今日用药洞察与建议
──────────────────────
{insights}
（基于今日数据，指出值得关注的用药规律，辅助医生调整策略）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

报告生成后，通过飞书发送给医生账号。
```

## 第九步：测试运行

### 9.1 单患者测试

先用单个患者测试整个流程：

1. 选取患者 P001（王建国，高血压）
2. 手动触发「随访调度器」，检查是否生成了正确的随访消息
3. 检查飞书是否收到了消息
4. 手动触发「患者回复模拟器」，生成一条回复
5. 检查「回复处理器」是否正确解析并写入了 followups.csv
6. 查看 chat_logs.csv 是否有完整记录

### 9.2 全量演示

测试通过后，开启全部30个患者的演示：
1. 启动「随访调度器」定时任务（每10分钟）
2. 启动「回复处理器」消息监听
3. 启动「患者回复模拟器」
4. 等待30分钟（= 系统内3天），观察第一批随访消息是否正常发出
5. 等待10分钟后检查第一份医生日报

### 9.3 常见问题排查

| 问题 | 可能原因 | 解决方法 |
|------|---------|---------|
| 飞书消息发不出去 | App ID/Secret 配置错误 | 重新检查飞书开放平台的凭证 |
| 消息发出但内容是模板 | Prompt 未读取到患者数据 | 检查 CSV 文件路径和字段名是否匹配 |
| 回复没有被解析 | 消息监听未正确配置 | 检查飞书 Webhook 配置 |
| CSV 文件写入失败 | 文件权限或路径问题 | 检查项目 `data/` 目录权限 |
| 定时任务没有触发 | 时区设置错误 | 确认 OpenClaw 时区设置为 Asia/Shanghai |
